本文档用于定义“每日记账”项目的产品方向、功能范围、技术方案候选、协作规则与后续决策方式。

从本文档创建开始，整个项目期间默认遵守以下原则：

1. 用户是非技术背景，无法提供实现细节时，由 Codex 负责先整理多个可选方案。
2. 任何重要开发决策，必须先给出方案、解释优缺点、说明推荐项，再由用户选择。
3. 在用户未确认前，不擅自推进高影响决策，例如：

5. 所有说明优先用通俗中文表达，减少术语；必须出现术语时，同时给出白话解释。

## 2. 产品目标

### 2.1 产品名称

每日记账

### 2.2 产品定位

一款运行在 Windows 系统上的个人记账应用，核心目标是帮助用户快速记录每一笔人民币支出，并按两级分类进行统计和回顾。

### 2.3 目标用户

- 个人用户
- 非财务专业用户
- 希望每天简单记账的人
- 希望在电脑上使用，而不是必须用手机的人

### 2.4 核心价值

1. 记录简单：尽量少步骤完成一笔支出录入。
2. 分类清晰：支持一级分类和二级分类。
3. 回顾方便：可以查看支出明细与分类统计。
4. 适合新手：界面直观，不要求用户理解技术概念。

---

## 3. 首版范围（MVP）

首版建议只做最重要、最容易用起来的功能。

### 3.1 必做功能

1. 新增一笔支出
2. 编辑一笔支出
3. 删除一笔支出
4. 记录字段：
   - 金额（人民币）
   - 支出日期
   - 一级分类
   - 二级分类
   - 备注
5. 查看支出列表
6. 按日期筛选
7. 按一级/二级分类筛选
8. 查看基础统计：
   - 今日支出
   - 本月支出
   - 各一级分类汇总
   - 各二级分类汇总

### 3.2 首版暂不做

1. 收入管理
2. 多用户账号
3. 云同步
4. 手机端同步
5. 发票识别
6. AI 自动分类
7. 预算预警
8. 银行卡/支付平台自动导入

这些功能未来都可以追加，但不建议一开始做太多，否则会拖慢落地速度。

## 4. 支出记录字段设计

每条支出记录建议包含以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | 系统生成 | 每条记录唯一编号 |
| amount | 数值 | 支出金额，单位为人民币元 |
| expense_date | 日期 | 实际花钱日期 |
| category_level_1 | 文本 | 一级大类 |
| category_level_2 | 文本 | 二级小类 |
| note | 文本 | 备注，可为空 |
| created_at | 日期时间 | 创建时间 |
| updated_at | 日期时间 | 最后修改时间 |

补充规则：

1. 金额必须大于 0
2. 金额默认保留 2 位小数
3. 分类必须先选一级，再选对应二级
4. 备注可留空

## 6. 界面与使用流程建议

### 6.1 首版页面结构

建议首版包含 4 个主要界面：

1. 首页/总览
   - 今日支出
   - 本月支出
   - 最近记录
   - 分类汇总卡片
2. 新增记录页
   - 快速录入一笔支出
3. 明细列表页
   - 查看、筛选、编辑、删除
4. 分类统计页
   - 按一级和二级分类统计

### 6.2 单笔录入推荐流程

1. 输入金额
2. 选择日期
3. 选择一级分类
4. 选择二级分类
5. 输入备注（可选）
6. 点击保存

### 6.3 易用性要求

1. 默认打开就能快速新增记录
2. 常用按钮文字清楚，例如“新增支出”“保存记录”
3. 避免出现复杂术语
4. 删除操作要有二次确认

### 7.2 非技术用户友好目标

1. 尽量生成可直接双击运行的桌面应用

## 8. 技术方案候选

下面是目前最适合这个项目的 3 个技术路线。最终已由用户确认选择方案 A。

### 方案 A：桌面客户端（推荐）

**建议组合**

- 前端界面：React
- 桌面壳：Tauri
- 本地数据库：SQLite

**白话解释**

这是一个真正的 Windows 桌面软件。打开后像普通电脑软件一样使用，数据存储在本机。

**优点**

1. 最符合“Windows App”的直觉
2. 运行速度快，占用资源相对低
3. 本地 SQLite 很适合记账这类结构化数据
4. 后续可打包成安装程序，交付体验较好
5. 后续扩展统计、导出、备份都比较自然

**缺点**

1. 开发复杂度高于普通网页
2. 打包、安装、版本升级流程会更复杂一些
3. 需要处理桌面应用和前端的组合工程

**适合你当前项目的原因**

1. 你明确要求 Windows 可用
2. 记账数据适合本地数据库管理
3. 未来如果要长期使用，这个路线更稳

---

### 方案 B：本地网页应用

**建议组合**

- 前端界面：React
- 运行方式：浏览器打开本地页面
- 数据存储：浏览器本地存储或 IndexedDB

**白话解释**

它本质上是一个网页，但只在你自己的电脑上使用。优点是开发快，但不像真正的软件。

**优点**

1. 开发最快
2. 界面灵活，容易迭代
3. 不需要复杂桌面打包
4. 前期验证产品想法成本最低

**缺点**

1. 更像网页，不像标准桌面软件
2. 数据保存在浏览器中，备份和迁移理解成本较高
3. 浏览器缓存、清理数据等操作可能让新手困惑
4. 后续若转成真正桌面应用，可能需要再改造

**适合场景**

1. 想先最快看到原型
2. 先验证功能，再决定是否做成桌面安装包

---

### 方案 C：Electron 桌面应用

**建议组合**

- 前端界面：React
- 桌面壳：Electron
- 本地数据库：SQLite

**白话解释**

也是桌面软件，成熟度很高，很多知名桌面应用都使用类似路线。

**优点**

1. 生态成熟，资料很多
2. 做桌面功能时经验和案例最多
3. 对前端开发友好
4. 打包 Windows 应用比较常见

**缺点**

1. 安装包通常更大
2. 内存占用通常高于 Tauri
3. 相比 Tauri，整体更“重”

**适合场景**

1. 更重视成熟生态
2. 不介意安装包体积更大
3. 希望减少桌面壳层面的新技术不确定性

## 9. 技术方案对比总结

| 方案 | 类型 | 开发速度 | 安装体验 | 性能/体积 | 数据管理 | 推荐程度 |
| --- | --- | --- | --- | --- | --- | --- |
| A | Tauri 桌面应用 | 中 | 好 | 好 | 好 | 高 |
| B | 本地网页应用 | 高 | 一般 | 好 | 一般 | 中 |
| C | Electron 桌面应用 | 中 | 好 | 一般 | 好 | 中高 |

### 当前推荐

优先推荐 **方案 A：React + Tauri + SQLite**。

推荐理由：

1. 更像真正的 Windows 软件
2. 本地数据库更适合长期记账
3. 性能、体积、可维护性整体更平衡

## 16. 首版分类能力确认结果

### 问题

首版是否支持用户自定义一级分类和二级分类。

### 用户确认结果

**方案 A：首版使用系统预置分类，不支持用户自定义分类**

### 影响

1. 首版只提供系统内置的一级分类和二级分类
2. 用户不能新增、编辑、删除分类
3. 录入支出时只能从预置分类中选择
4. 后续如果需要自定义分类，可作为二期功能追加

### 这样做的好处

1. 功能范围更小，更适合先完成可用版本
2. 分类逻辑更稳定，减少错误情况
3. 界面更简单，适合首次使用的用户

### 修复说明

1. 网页版运行时没有走 Tauri 本地命令

3. 所以在浏览器模式下无法真正写入数据

现在已修复为：

1. 桌面版使用 SQLite 本地数据库
2. 网页版使用浏览器本地存储
3. 两种运行方式都可以正常新增、编辑、删除并刷新统计
